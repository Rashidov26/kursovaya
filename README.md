Введение
Система дистанционного управления (РП) является основным интерфейсом между пилотом и беспилотным летательным аппаратом. От её надёжности и оперативности зависит безопасность полёта. В данном проекте реализован приём сигнала PPM (Pulse Position Modulation) от RC-приёмника с использованием аппаратного таймера STM32 в режиме Input Capture. Понимание принципов работы PPM, обработки прерываний и механизмов Fail-Safe необходимо для разработки отказоустойчивых систем управления БПЛА.

В курсовой работе рассматривается модуль Tx_and_Rx.ino, а также настройка таймеров (timer.ino). Цель – изучить архитектуру приёма команд, проанализировать алгоритм декодирования PPM, выполнить расчёты временных параметров и предложить улучшения.

1. Теоретические основы системы дистанционного управления
1.1. Форматы сигналов PWM и PPM
PWM (Pulse Width Modulation) – каждый канал передаётся по отдельному проводу. Длительность импульса от 1000 до 2000 мкс, период 20 мс. Недостаток – много проводов.

PPM (Pulse Position Modulation) – все каналы передаются последовательно по одному проводу. Кадр начинается синхроимпульсом длительностью >3000 мкс, затем идут импульсы каналов (1000–2000 мкс). Стандартный кадр для 8–10 каналов имеет период 20–27 мс. PPM экономит вес и упрощает подключение.

В проекте используется PPM, поступающий на вывод PA0 (TIM2 Channel 1). Таймер STM32F103 тактируется частотой 72 МГц, с предделителем PSC=71 получаем частоту счётчика 1 МГц (1 тик = 1 мкс). Режим Input Capture позволяет фиксировать значение счётчика в момент фронта сигнала без участия процессора, что даёт точность до 1 мкс.

1.2. Принцип работы обработчика PPM
На каждый нарастающий фронт сигнала PPM вызывается прерывание. В обработчике вычисляется длительность последнего импульса как разность между текущим захваченным значением CCR1 и предыдущим. Если происходит переполнение 16-битного счётчика (0xFFFF → 0), разность корректируется добавлением 0xFFFF. Далее, если импульс >3000 мкс – это синхроимпульс, сбрасывается счётчик каналов. Если в диапазоне 800–2200 мкс – сохраняется в соответствующий канал (channel_1 … channel_10).

1.3. Таймеры STM32
Таймер TIM2 настроен как счётчик с периодом 0xFFFF (65535 тактов). Частота счётчика: f_timer = 72 МГц/(71+1) = 1 МГц, период тика = 1 мкс. Максимальное измеряемое время до переполнения – 65535 мкс ≈ 65,5 мс, что перекрывает период PPM-кадра (до 27 мс).

2. Анализ исходного кода
Основные файлы: timer.ino (настройка TIM2 и TIM4) и Tx_and_Rx.ino (обработчик прерывания PPM). Ниже представлен аннотированный листинг ключевых фрагментов.

2.1. Настройка таймера TIM2 (timer.ino)
void timer_setup() {

    // TIM2 – Input Capture на PA0 (PPM)

    TIMER2_BASE->CR1 = TIMER_CR1_CEN;   // включить счётчик

    TIMER2_BASE->PSC = 71;              // предделитель: 72МГц/(71+1)=1МГц

    TIMER2_BASE->ARR = 0xFFFF;          // счёт до 65535

    // Канал 1 – вход захвата TI1

    TIMER2_BASE->CCMR1 = TIMER_CCMR1_CC1S_INPUT_TI1;

    TIMER2_BASE->CCER = TIMER_CCER_CC1E;      // разрешить захват

    TIMER2_BASE->CCER &= ~TIMER_CCER_CC1P;    // захват по нарастающему фронту

    TIMER2_BASE->DIER = TIMER_DIER_CC1IE;     // прерывание по захвату

    Timer2.attachInterrupt(1, handler_channel_1);

}

2.2. Обработчик прерывания PPM (Tx_and_Rx.ino)
volatile int32_t measured_time, measured_time_start;

volatile uint8_t channel_select_counter = 0;

void handler_channel_1() {

    // 1) Вычисляем длительность импульса

    measured_time = TIMER2_BASE->CCR1 - measured_time_start;

    // 2) Коррекция переполнения счётчика

    if (measured_time < 0) measured_time += 0xFFFF;

    // 3) Запоминаем текущее время

    measured_time_start = TIMER2_BASE->CCR1;

    // 4) Определяем тип импульса

    if (measured_time > 3000) {

        channel_select_counter = 0;        // синхроимпульс -> начало кадра

    } else if (measured_time >= 800 && measured_time <= 2200) {

        channel_select_counter++;

        switch(channel_select_counter) {

            case 1: channel_1 = measured_time; break;  // Roll

            case 2: channel_2 = measured_time; break;  // Pitch

            case 3: channel_3 = measured_time; break;  // Throttle

            case 4: channel_4 = measured_time; break;  // Yaw

            case 5: channel_5 = measured_time; break;  // Flight Mode

            // ... до channel_10

        }

    }

}

Пояснения: Переменные объявлены как volatile – это запрещает компилятору оптимизировать их кэширование, так как они изменяются в прерывании. Обработчик должен быть максимально коротким, чтобы не блокировать другие прерывания и главный цикл.

4. Расчётная часть (выполнение заданий варианта 11)
Задание 11.1 (базовый уровень) – Форматы RC-сигналов и временные параметры
а) Сравнение PWM и PPM (таблица приведена в теоретической части).

б) Период PPM-кадра для 10 каналов. Максимальная длительность канального импульса = 2000 мкс, синхроимпульс = 4000 мкс.

T_frame = Nch · tmax + tsync = 10·2000 + 4000 = 24000 мкс = 24 мс.

в) Частота обновления каналов (частота кадров):

fRC = 1 / T_frame = 1 / 0.024 ≈ 41,67 Гц.

Вывод: При 10 каналах максимальная частота управления – около 42 Гц, что достаточно для пилотирования, но для гоночных дронов применяют более быстрые цифровые протоколы.

Задание 11.3 – Анализ обработчика и переполнения счётчика
б) Коррекция переполнения: 16-битный счётчик считает от 0 до 65535. Если между двумя фронтами произошло переполнение, то разность CCR1 - measured_time_start станет отрицательной. Добавление 0xFFFF (65535) восстанавливает истинную длительность. Например: старый счётчик = 65000, новый = 1000 (после переполнения). Разность = 1000-65000 = -64000; +65535 = 1535 мкс – корректная длительность.

в) Ключевое слово volatile: Без volatile компилятор может закешировать значение переменной в регистре, и в основном цикле будет читаться устаревшее значение. Для переменных, изменяемых в прерывании, volatile обязателен.

г) Максимальная задержка от движения стика до обновления channel_1: Задержка складывается из времени до следующего синхроимпульса (в худшем случае – почти целый кадр 24 мс) плюс время до конкретного канала (до 2000 мкс). Итого ~26 мс. В основном цикле latenc добавляет до 4 мс ожидания синхронизации, но значение channel_1 будет прочитано в следующей итерации после установки.

Задание 11.4 – Fail-Safe
а) Реализация: В коде проекта (Flight_Controller.ino) в функции check_failsafe``(``) отслеживается отсутствие новых кадров PPM по таймеру. Если в течение 1 секунды не было ни одного синхроимпульса (переменная new_frame не обновлялась), то активируется Fail-Safe: channel_3 (газ) устанавливается в 1000 мкс (моторы остановлены), остальные каналы – в нейтральное положение 1500 мкс, flight_mode принудительно переводится в режим посадки.

в) Временная диаграмма: нормальная работа → потеря сигнала → через 1 с срабатывание Fail-Safe (газ=1000, дрон снижается) → восстановление сигнала → нормальное управление.

г) Улучшенный Fail-Safe (предложение): вместо немедленной остановки моторов – инициировать алгоритм RTH (возврат домой), если есть GPS-фикс. Если GPS нет – плавно снижаться со скоростью 0,5 м/с до касания. Это требует модификации – добавление проверки высоты и управления газом через ПИД высоты.

Задание 11.5 – Цифровые протоколы RC
а) Сравнительная таблица протоколов:

Протокол |Тип |Каналы |Латентность |Шифрование

PPM |Аналог |8-10 |15-27мс |Нет

SBUS |Цифровой |16 |7мс |Нет

CRSF(Crossfire) |Цифровой |16 |<2мс |Нет

ELRS |Цифровой |16 |<1мс |Да(AES)

б) Преимущество цифровых протоколов: они используют более высокую скорость UART (до 400 кбит/с), кадр передаётся быстрее, нет необходимости в длинных синхроимпульсах. Кроме того, цифровые протоколы имеют проверку CRC и меньше подвержены помехам.

в) Поддержка SBUS вместо PPM: потребуется изменить настройку UART (Serial) на инвертированный сигнал (SBUS инвертирован), приём данных по прерыванию UART, парсинг пакетов 25 байт с фиксированной структурой. Таймер Input Capture уже не нужен – используется UART + DMA.

5. Предложение по улучшению системы управления
В текущей реализации Fail-Safe останавливает моторы, что при полёте над препятствиями или водой приведёт к аварии. Предлагается интеграция с алгоритмом RTH:

При потере сигнала на 1 секунду активируется режим RTH (flight_mode=4), если есть валидные GPS-координаты домашней точки.
Дрон набирает безопасную высоту 15 м (если ниже) и возвращается к точке взлёта.
Если GPS недоступен – включается плавное снижение с использованием лидара/барометра до касания земли.
Реализация: добавить в функцию check_failsafe() вызов RTH и установку соответствующих флагов. Потребуется также контроль времени выполнения RTH и таймаут на случай отказа GPS.
Сложность реализации: средняя, потребуется модифицировать несколько модулей (Telemetry для индикации, RTH.ino). Это повысит безопасность полёта до уровня, требуемого стандартами (например, DO-178C уровень DAL-C).

Проведён детальный анализ PPM-сигнала, алгоритма декодирования через аппаратный таймер в режиме Input Capture. Выполнены расчёты временных параметров (период кадра 24 мс, частота обновления 42 Гц).
